# Git 仓库管理规范

## 1. 目标

这份规范用于管理公共资产仓库的结构、提交和发布节奏，避免公开资产越积越多后失控。

## 2. 仓库定位

- `editorial/asset-policy.md`：资产管理规范
- `editorial/repo-management.md`：公共仓库管理规范

本仓库不负责私有大纲、文章篇号或写作流程本身。

## 3. 允许内容

- 共享图示与可编辑源文件
- 公共 Prompt
- 公共 schema
- 公共评测集与 Judge
- 公共示例代码
- 公共参考资料索引

## 4. 禁止反向依赖私有仓库

- 不要在本仓库的必读规则里要求先读取私有仓库文件
- 不要把只能在私有语境下理解的中间材料放进本仓库
- 如果某项资产必须依赖私有文章草稿才能成立，先不要公开

## 5. 目录约定

- 共享资产优先放到 `assets/`
- Prompt、schema、eval、example、reference 尽量各归其位
- 根目录尽量只保留高价值入口文件，不继续堆散文件

## 6. 分支建议

如果需要分支工作，推荐使用：

- `asset/<topic>`：图示、数据、评测资产整理
- `prompt/<topic>`：Prompt 与 Judge 资产
- `schema/<topic>`：工具协议与接口定义
- `ops/<topic>`：仓库结构、脚本、规范维护

如果只是快速迭代，也可以直接在 `main` 上工作，但要保持提交粒度清晰。

## 7. 提交信息建议

推荐使用前缀，让历史更可读：

- `asset:` 图片、数据、trace 等资产
- `eval:` 评测集、Judge、报告
- `prompt:` Prompt 资产
- `schema:` 工具 schema
- `editorial:` 写作规范、发布规范
- `repo:` 仓库结构与基础设施

示例：

- `asset: add agent loop diagram sources`
- `repo: scaffold asset and eval directories`

## 8. Tag 建议

如果公共资产对外发布节奏稳定，建议按资产包或阶段打 tag。

推荐格式：

- `public/v1`
- `public/v2`
- `asset/<topic>/v1`

这样可以反查：

- 某次公开时资产是什么版本
- 某次公开时 Prompt、评测配置是什么版本

## 9. 提交前检查

提交前至少确认：

- 没有把敏感数据直接放进仓库
- 没有只提交导出图而漏掉源码
- 没有把临时文件和中间产物混进提交
- 新增文件是否放到了最合理的目录
- 有没有意外泄露私有写作痕迹、内部讨论或未发布正文

## 10. 不建议做的事

- 不要把大量新文件继续扔在根目录
- 不要把同一资产在多个地方重复存放
- 不要无说明地覆盖重要旧版本
- 不要把私有仓库路径写成这里的强依赖

## 11. 当前迁移策略

本仓库当前采用“从私有仓库拆分出来”的增量整理方式：

- 先承接可公开目录
- 再逐步补齐真正可对外复用的资产
- 不急于把所有内部材料都公开化

这样比一次性公开全部内容更稳，也更不容易踩隐私和语境边界。
