# 资产管理规范

## 1. 目标

资产管理的目标不是把文件分类得很漂亮，而是保证下面三件事：

- 重要素材不会丢
- 以后还能编辑，而不只是重新截图
- 文章、评测、示例之间能共享资产

## 2. 什么算重要资产

以下内容默认都算重要资产：

- 图示源码，例如 `drawio`、`mermaid`、`excalidraw`
- 导出图，例如 `png`、`svg`
- 文章中反复复用的截图
- 示例数据、脱敏数据、评测样本
- Prompt
- Function Calling schema
- MCP 能力定义
- Judge Prompt 与评测配置
- trace 样例与关键错误复盘材料

## 3. 留存原则

### 3.1 先留源码，再留导出物

错误做法：

- 只留一张 `png`
- 下次改图时重新画

正确做法：

- 把可编辑源文件放到 `assets/diagrams/src/`
- 把导出文件放到 `assets/diagrams/exports/`

### 3.2 重要资产不要只存在文章正文里

例如：

- Prompt 不应只出现在文章段落中
- schema 不应只藏在代码块里
- 评测数据不应只存在截图里

应该把它们拆出来，放到对应目录中单独管理。

### 3.3 公共资产优先共享存放

如果多个文章会反复用到同一张图、同一份 Judge Prompt 或同一种 schema：

- 优先放共享目录
- 文章正文通过路径引用

## 4. 命名规范

优先使用 `slug` 驱动命名，而不是强绑定篇号。

推荐格式：

`<article-slug>__<asset-name>__v1.<ext>`

例如：

- `agent-not-chatbot__agent-loop__v1.drawio`
- `when-not-to-use-agent__decision-matrix__v1.png`
- `tool-design__judge-prompt__v2.md`

## 5. 数据与隐私

### 5.1 严禁直接入库的内容

- 真实用户隐私数据
- 企业内部订单、工单、日志原文
- 含密钥、账号、路径信息的截图或 trace

### 5.2 可以入库的内容

- 人工构造样本
- 已充分脱敏的结构化样本
- 只保留结构、不保留真实身份信息的 trace

## 6. 大文件策略

- 小文件直接 Git 管理
- 大于 `10MB` 的长期资产，优先考虑 `Git LFS`
- 不要同时提交多个重复导出版本
- 对临时中间产物，优先不入库

## 7. 删除与替换

不要轻易覆盖旧版重要资产。

更推荐的做法：

- 保留版本号
- 新旧并存一段时间
- 确认文章和引用都切换后，再做清理

## 8. 最终判断标准

一个资产管理方案是否合格，只看这三个问题：

1. 半年后还能不能找到它
2. 半年后还能不能改它
3. 半年后别人能不能看懂它是干什么的
